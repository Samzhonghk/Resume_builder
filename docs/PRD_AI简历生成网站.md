# AI 简历生成网站 — 产品需求文档（PRD）

## 1. 产品概述

一个面向新西兰求职者的 AI 简历生成与优化工具，支持两种创建路径（从零填写 / 上传旧简历优化），覆盖所有职业方向，并提供配套的求职信（Cover Letter）生成功能。

**核心价值主张**：让用户用更少的时间，产出符合新西兰本地习惯、ATS（自动筛选系统）友好、且针对目标行业优化措辞的简历。

**目标用户**：在新西兰求职的人群，覆盖所有职业类型，不限于 IT/AI 行业。

---

## 2. 核心用户流程

```
落地页
  │
  ├─ 入口A: 从零开始创建
  │     → 空白表单 → 用户填写经历 → AI生成完整文案
  │
  └─ 入口B: 上传旧简历
        → 上传文件(PDF/Word) → AI提取结构化数据 → 回填表单(预填充) → 用户确认/修改 → AI优化润色
                                                                              │
        ┌─────────────────────────────────────────────────────────────────┘
        ↓
  模板选择(实时预览) → 导出(PDF)
        ↓
  [可选] 生成配套 Cover Letter(粘贴目标JD) → 关键词匹配度检查
```

**关键设计原则**：两条入口路径最终汇合到同一个表单编辑器和同一套数据结构，只是表单的初始状态（空 vs 预填充）和后续 AI 调用的 prompt 类型（生成 vs 优化）不同。底层组件只需开发一套。

## 3. 功能模块详细说明

### 3.1 落地页

- 两个清晰的 CTA 入口：「从零开始创建」「上传简历，快速优化」
- 简要说明产品定位（通用职业，新西兰本地化）

### 3.2 编辑器（核心模块）✅ 已完成（阶段一）

**布局**：双栏布局，左侧表单 / 右侧实时预览（用户编辑表单时，右侧同步渲染当前模板下的效果）

**表单字段结构**（两条路径共用）：
- 基本信息：姓名、联系方式（注意新西兰电话格式 `+64`，不需要照片字段）
- **目标行业/职位类型**（新增字段，见 3.5）
- 工作经历：公司、职位、起止时间、要点描述（多条 bullet）
- 教育背景
- 技能列表
- 其他可选板块（证书、语言、志愿经历等）

**数据结构**（两条路径统一，已通过代码审计确认为当前实现的标准结构）：
```json
{
  "source": "blank" | "uploaded",
  "targetIndustry": "string",
  "data": {
    "name": "...",
    "contact": {...},
    "workExperience": [
      {"company": "...", "title": "...", "period": "...", "bullets": ["...", "..."]}
    ],
    "education": [...],
    "skills": [...]
  }
}
```

> **重要约束**：这套数据结构是后续所有路径和模板的共同基础。上传优化路径解析出的数据、新增模板的渲染输入，都必须严格对齐这一结构，不另起新结构。

### 3.3 上传与提取模块（入口B专属）

**支持格式**：PDF、Word（.docx）

**处理流程**：
1. 文件解析：PDF 用 `pdf-parse`，Word 用 `mammoth.js`，提取原始文本
2. 结构化提取：将原始文本交给 LLM，按统一 JSON 格式输出结构化简历数据（不要用正则规则解析，简历格式差异太大，交给 LLM 做信息抽取更稳健）
3. **回填表单 + 确认/编辑中间页**：提取结果填入编辑器表单，渲染为独立的「确认/编辑」中间页面，**用户必须在此页确认或修改后才能进入下一步**（AI 提取可能出错，需要人工校验环节）
4. 数据落地：用户确认后的数据，**严格对齐 3.2 中定义的统一简历数据结构**，与表单创建路径共用同一套模板渲染、AI优化、导出组件，不为上传路径单独维护一套

### 3.4 AI 文案生成/优化模块

根据 `source` 字段分支选择 prompt：

- **`source === "blank"`**（生成模式）：
  > 根据用户填写的经历要点，生成完整简历文案，使用 STAR 法则（情境-任务-行动-结果）增强表达力。

- **`source === "uploaded"`**（优化模式）：
  > 基于用户已有的简历内容优化润色，使其更符合目标行业的简历惯例，**不得编造用户未提供的具体事实或数据**（防止简历造假风险）。

**通用约束**（两种模式都要遵守）：
- 符合新西兰简历本地习惯：不放照片、不放年龄/婚姻状况、日期格式按当地习惯、篇幅控制在 1-2 页
- ATS 友好：避免依赖图片/图标传递关键信息

### 3.5 行业适配机制

为兼顾"覆盖所有职业"与"AI 优化质量"，采用**预设大类 + 自定义**的折中方案：

表单中"目标行业"字段为下拉选择：
- 软件工程 / IT / AI
- 数据 / 分析类
- 市场营销 / 销售
- 医疗 / 护理
- 教育
- 商业 / 金融
- 其他（自由文本输入）

每个预设大类后台对应一段**该行业简历惯例的简短指引文本**（例如：软件工程类强调项目成果与技术栈、用量化数据；护理类强调资质认证与合规；销售类强调业绩数字），生成/优化时将对应指引拼入 prompt。选择"其他"时退化为通用 prompt。

> 这一機制让产品同时服务广泛职业群体，又不至于让 AI 优化变得千篇一律。

### 3.6 模板系统

**模板清单（已更新）**：

| 模板名 | 状态 | 文件名 |
|---|---|---|
| ClassicTemplate（经典单栏） | ✅ 已完成 | ClassicTemplatePDF.jsx |
| ModernTemplate（蓝色强调） | ✅ 已完成 | ModernTemplatePDF.jsx |
| MinimalTemplate（极简无分隔） | ✅ 已完成 | MinimalTemplatePDF.jsx |
| ProfessionalTemplate（居中页眉） | ✅ 已完成 | ProfessionalTemplatePDF.jsx |
| CreativeTemplate（深色页眉块） | ✅ 已完成 | CreativeTemplatePDF.jsx |

> **变更说明**：原计划中的"双栏简洁"模板暂缓。本轮新增的 4 个模板均**完全照搬 ClassicTemplate 的代码模式**（纯 react-pdf 原语手刻，单栏布局），不引入双栏或新的渲染方案。若后续仍需要双栏布局模板，需作为独立技术任务评估（react-pdf 手刻双栏的实现复杂度明显更高）。

**设计约束（硬性要求，所有模板统一遵守）**：
- 风格中性，不偏向任何单一行业（避免"看起来像程序员专用"）
- ATS 兼容：避免复杂多栏表格、避免非标准字体（用 Arial/Calibri 等系统常见字体）、避免用图片代替文字信息
- 与对应的 HTML 预览版本保持视觉一致（已验证的实现方式：react-pdf 原语 Document / Page / View / Text + 内置 PDF 标准字体 Helvetica 系列）

**技术实现**：模板为独立的 React 渲染组件，统一吃同一份 JSON 数据，切换模板不需重新处理数据：
```jsx
<ResumePreview data={resumeData} template="classic" />
<ResumePreview data={resumeData} template="modern" />
<ResumePreview data={resumeData} template="minimal" />
<ResumePreview data={resumeData} template="professional" />
<ResumePreview data={resumeData} template="creative" />
```

**交互方式**：模板选择页直接用用户当前的真实简历内容做实时预览，不用空白样例图。

### 3.7 导出模块 ✅ 已完成（阶段一，Classic 模板）

- 导出方案确定为 `react-pdf`（已验证可行，体积约 1.7MB，MVP 阶段可接受）
- 已实现：`handleExportPDF` 调用 `pdf(<ClassicTemplatePDF />).toBlob()` 生成 PDF，触发浏览器下载，文件名为 `{姓名}-resume.pdf`
- 桌面端（≥1 列）：Download PDF 按钮在预览面板右上角；移动端（窄屏）：表单下方出现全宽按钮
- 已知踩坑：`@react-pdf/renderer` 不能静态引入，需在 `handleExportPDF` 内动态 `import`，否则 Vite 预构建出错（当前实现已采用此方案，`vite.config.js` 无需额外配置）

> 新增模板（3.6）需复用同一套导出逻辑（`ExportButton` / `handleExportPDF` 的 `full` prop 模式），不另起新的导出流程。

### 3.8 Cover Letter 生成模块 ✅ 已完成（阶段三）

**位置**：简历导出完成后的附加步骤，入口如「为这份简历生成求职信」

**数据复用**：复用统一简历数据结构中的「个人信息」部分（姓名、联系方式等），不重复采集。

**交互流程**：
1. 自动带入已生成的简历内容（不重复填写）
2. 用户粘贴目标岗位 JD（文本框）
3. 可选填公司名/岗位名称
4. 生成针对性求职信

**支持多份生成**：同一份简历可针对不同岗位 JD 生成多份不同的 cover letter（不是一份简历只能配一份信）

**Prompt 要点**：
> 突出简历中与该岗位最相关的经历，语气专业不生硬，符合新西兰求职信常见格式（开头说明申请意向，中间1-2段呼应岗位要求与自身经历匹配点，结尾表达沟通意愿）

**署名**：求职信末尾署名默认取简历中的 `name` 字段；若 `name` 为空，默认回退到 `Sam Zhong`。

**下载格式**：生成后支持 3 种格式下载（下拉菜单选择）：

| 格式 | 扩展名 | 实现方式 |
|---|---|---|
| PDF | `.pdf` | `@react-pdf/renderer` 动态 import，`CoverLetterPDF` 组件渲染 |
| Word / RTF | `.rtf` | 纯客户端生成 RTF 文本（无额外依赖），Word / LibreOffice / Google Docs 均可打开 |
| 纯文本 | `.txt` | Blob + `text/plain` |

> **技术选型说明**：Word 格式采用 RTF 而非 .docx，原因是 `docx` npm 包（4 MB）与 react-pdf / pdfjs-dist 叠加后在 Vite 8（rolldown/Rust）构建时导致 OS 级页面文件耗尽（exit code 9），而 RTF 为纯文本格式，无需额外依赖，已验证 Word、LibreOffice、Google Docs 均可正常打开编辑。

### 3.9 关键词匹配度检查（与 Cover Letter 模块同步）

**数据复用**：同样复用统一简历数据结构中的「个人信息」部分，与 Cover Letter 模块保持一致。

**功能逻辑**：将用户简历内容与粘贴的 JD 做关键词比对。由于产品覆盖所有行业、无法预设关键词库，**此功能必须基于用户粘贴的 JD 动态比对**，而非依赖预设关键词列表。

**输出形式**：以**高亮列表**形式展示 JD 中出现、但用户简历中缺失或不够突出的关键词，供用户参考补充，不直接代写或自动改写简历内容。

## 4. 明确不做的功能

**不做 Seek.co.nz 自动投递功能**。原因：
- 违反招聘平台服务条款（禁止自动化批量操作账号）
- 触发平台异常行为检测可能导致账号被封，反而损害求职者利益

替代方案：本产品聚焦于"投递前准备"（定制化简历 + 针对性 cover letter + 关键词检查），实际提交投递的动作由用户自己手动完成。

## 8. 技术债与部署

### API Key 安全（BFF 迁移）✅ 已完成

**原问题**：MVP 阶段 `VITE_ANTHROPIC_API_KEY` 写入前端环境变量，构建产物中 API Key 明文暴露于浏览器。

**已解决**：Claude API 调用已迁移至 Vercel Serverless Functions（`api/` 目录），前端通过 `fetch('/api/...')` 调用，Anthropic Key 只存在于服务端环境变量，浏览器无法获取。

**架构变化**：

| 文件 | 变化 |
|---|---|
| `api/generate.js` | 服务端处理简历 AI 优化，调用 Claude API |
| `api/extract.js` | 服务端处理上传简历的结构化提取 |
| `api/cover-letter.js` | 服务端处理求职信生成 |
| `api/keywords.js` | 服务端处理关键词缺口检查 |
| `src/services/*.js` | 改为调用 `/api/*` 接口，不再直接调用 Anthropic SDK |

**部署平台**：Vercel（免费 Hobby 计划）

**本地开发**：`npm run dev`（即 `vercel dev`）同时启动 Vite 开发服务器和 serverless functions

**环境变量**：
- 本地：`.env` 中设置 `ANTHROPIC_API_KEY=sk-...`
- Vercel 生产：在 Dashboard → Settings → Environment Variables 中设置
